3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1 Загальні відомості про систему
     Назва системи: Слеш Ліб (SlashLIB)
     Розробник: Команда екосистеми *Lib
     Призначення системи: SlashLIB є частиною глобальної екосистеми *Lib, яка включає в себе найбільші портали для читання манги, манхви та маньхуа. Система була виокремлена в окрему платформу для кращої структуризації специфічного контенту та надання користувачам спеціалізованого функціоналу. Платформа працює як повноцінне PWA (Progressive Web App), забезпечуючи доступ до контенту через браузер та мобільні застосунки.
     Ключовою особливістю системи є робота з великими обсягами графічних даних. Кожен тайтл (комікс) складається з сотень розділів, кожен з яких містить десятки зображень високої роздільної здатності. Це породжує складні інженерні виклики щодо швидкості завантаження, зберігання та доставки контенту кінцевому користувачу, незалежно від його географічного положення.
     Основними користувачами системи є дві великі групи: кінцеві споживачі (читачі), які взаємодіють із платформою для пошуку та читання тайтлів, та контент-мейкери (команди перекладачів/сканлейтери), які завантажують нові розділи та керують релізами.
     Функціональні можливості системи охоплюють не лише доставку графічного контенту, але й складні механізми: відстеження прогресу читання, формування персоналізованих стрічок оновлень (закладки), розширену систему тегів і фільтрації, а також соціальну взаємодію (коментарі, рейтинги, відгуки). Оскільки SlashLIB функціонує як PWA, важливою вимогою є підтримка часткового офлайн-доступу за допомогою Service Workers, що дозволяє зберігати завантажені розділи для читання без підключення до мережі.
     3.2 Архітектурний стиль системи
     SlashLIB є яскравим прикладом сучасної програмної системи, побудованої на основі мікросервісного архітектурного стилю з чітко вираженим АРІ-орієнтованим підходом. На відміну від класичної монолітної архітектури, такий підхід передбачає глибоку декомпозицію загальної функціональності на набір слабко зв'язаних, автономних сервісів. Кожен такий модуль реалізує окрему бізнес-функцію і може розроблятися, тестуватися та розгортатися незалежно від інших. Наприклад, сервіс керування закладками (Bookmark Service) та сервіс доставки сторінок (Reader Service) існують ізольовано, що дозволяє їм оновлюватися та масштабуватися незалежно один від одного в залежності від поточного навантаження. Обґрунтування вибору саме мікросервісної архітектури для SlashLIB полягає у високих вимогах до гнучкості, відмовостійкості та технологічної різноманітності. Використання ізольованих мікросервісів дає можливість архітекторам гнучко обирати оптимальний технологічний стек та тип бази даних для кожної конкретної задачі (принцип polyglot persistence). Оскільки система SlashLIB перебуває у стані постійного розвитку та стикається зі зростаючим потоком користувачів, такий підхід забезпечує максимально високу адаптивність до нових вимог ринку. Це дозволяє команді розробників безперервно впроваджувати інновації та новий функціонал без ризику порушити стабільну роботу всієї платформи. 
     3.3 Основні компоненти системи
     На високому рівні архітектура SlashLIB складається з трьох основних шарів: рівень представлення (Frontend), рівень бізнес-логіки (Backend Microservices) та шар інфраструктури (Databases, S3, CDN). Взаємодія між клієнтом та бекендом відбувається через єдину точку входу — API Gateway, який відповідає за маршрутизацію запитів, автентифікацію та балансування навантаження.

Рисунок 3.1 — Високорівнева архітектура та декомпозиція SlashLIB
     Frontend-частина системи побудована за принципом SPA (Single Page Application), що забезпечує плавний досвід читання без перезавантаження сторінок. Backend декомпонований на функціональні модулі: User Service (профілі), Manga Catalog (метадані творів), Reader Service (доставка сторінок) та Social Service (коментарі та рейтинги).

Рисунок 3.2 — Схема функціональних модулів системи
     Внутрішня логіка бекенду чітко розділена на ізольовані домени:
     - User Service (Сервіс користувачів): Відповідає за реєстрацію, автентифікацію (з використанням JWT-токенів) та управління профілями. Цей сервіс є критичним для безпеки, але має відносно низьке навантаження порівняно з іншими модулями.
     - Catalog & Search Service (Сервіс каталогу): Керує метаданими коміксів (назви, автори, описи). Тісно інтегрований з Elasticsearch для забезпечення миттєвого повнотекстового пошуку та складної фільтрації за десятками жанрів і тегів.
     - Reader Service (Сервіс читалки): Найбільш навантажений вузол системи. Він відповідає за формування черги сторінок для конкретного розділу, генерацію тимчасових посилань на зображення та відстеження прочитаних сторінок.
     - Progress & Notification Service: Асинхронно обробляє прогрес користувача (на якій сторінці він зупинився) та розсилає push-сповіщення про вихід нових розділів для тайтлів, що знаходяться у закладках.
     SlashLIB використовує стратегію поліглотного зберігання даних (Polyglot Persistence), що дозволяє використовувати найбільш ефективні типи баз даних для специфічних задач. Реляційні дані про користувачів та команди перекладачів зберігаються в PostgreSQL, що забезпечує цілісність даних. Для складного та швидкого пошуку за десятками тегів використовується Elasticsearch. Кешування популярних тайтлів на головній сторінці реалізовано через Redis, що критично важливо для зниження навантаження на основну БД.
     Самі зображення сторінок манги зберігаються у хмарному об'єктному сховищі S3 і доставляються користувачам через мережу CDN (Content Delivery Network), що мінімізує затримки при читанні.

Рисунок 3.3 — Схема поліглотного зберігання даних
     3.4 Взаємодія компонентів
     Взаємодія між сервісами відбувається як синхронно (REST, gRPC), так і асинхронно (RabbitMQ). Наприклад, при читанні манги клієнт робить прямі REST-запити до Reader API. Проте, коли команда перекладачів завантажує новий розділ, система ініціює ланцюжок асинхронних подій для стиснення зображень, створення прев'ю та розсилки сповіщень читачам.

Рисунок 3.4 — Діаграма послідовності взаємодії сервісів
     Взаємодія в системі будується на комбінації синхронних та асинхронних патернів. Основний потік читання виглядає так: клієнт надсилає запит до API Gateway, авторизуючись за допомогою JWT-токена. API Gateway перенаправляє запит до Reader Service. Для мінімізації навантаження на основні сервери, Reader Service не віддає самі зображення; він повертає клієнту масив попередньо підписаних URL-адрес (Pre-signed URLs) або прямих посилань на CDN (Cloudflare). Після цього клієнтський браузер починає асинхронно завантажувати зображення безпосередньо з найближчого edge-сервера CDN (lazy loading).
     Водночас кожна перегорнута сторінка генерує фонову подію (наприклад, через WebSocket або фонові POST-запити), яка потрапляє у чергу повідомлень (RabbitMQ). Ці події асинхронно обробляються сервісом прогресу для оновлення статусу "прочитано", що дозволяє не блокувати інтерфейс користувача під час читання (Fail-fast підхід).

     3.5 Масштабованість та надійність
     Система розроблена з урахуванням горизонтального масштабування. При зростанні трафіку кількість екземплярів API-серверів збільшується автоматично за допомогою оркестрації в Kubernetes. Для забезпечення відмовостійкості використовується реплікація баз даних та механізми Failover. Навіть у разі відмови одного з регіонів хмарного провайдера, система продовжує працювати завдяки розподіленій інфраструктурі.
     Архітектура SlashLIB спроектована з урахуванням специфіки споживання графічного контенту — так званих "спайків" (різких стрибків трафіку), які виникають у момент релізу очікуваних розділів популярних тайтлів. Для боротьби з цим на рівні API Gateway застосовується агресивне обмеження частоти запитів (Rate Limiting) для захисту від парсерів та скраперів.
     Горизонтальне масштабування реалізовано за допомогою метрик Kubernetes: при досягненні порогового значення використання CPU або оперативної пам'яті (HPA - Horizontal Pod Autoscaler) система автоматично піднімає нові репліки Reader Service та Catalog Service. Кешування популярних запитів у Redis (патерн Cache-Aside) дозволяє переживати пікові навантаження без падіння основної бази PostgreSQL. У разі недоступності основного дата-центру хмарного провайдера трафік автоматично перемикається на резервний регіон (Failover-стратегія).
       
     Контейнеризація за допомогою Docker дозволяє уніфікувати середовище розробки та продуктивної експлуатації. Всі мікросервіси розгортаються незалежно, що дозволяє проводити Continuous Deployment без зупинки всього сервісу.

Рисунок 3.5 — Діаграма розгортання в хмарному середовищі
     3.6 Приклади програмної реалізації
     Під час виконання практичного заняття та проектування архітектури системи SlashLIB було використано інструмент штучного інтелекту (Antigravity на базі Gemini 3 Flash).
     Метою залучення ШІ-асистента було отримання базових шаблонів програмного коду, генерування структурних схем взаємодії мікросервісів, а також уточнення специфічних архітектурних патернів (таких як Polyglot Persistence та механізми горизонтального масштабування). ШІ допоміг у первинному структуруванні звіту згідно з вимогами ДСТУ та підборі коректної технічної термінології.
     Особистий внесок автора полягає у критичному аналізі згенерованих ШІ матеріалів та їх глибокій адаптації до специфіки предметної області — високонавантаженої платформи для читання манги. Автор самостійно:
     - визначив та обґрунтував вибір мікросервісного архітектурного стилю;
     - спроектував високорівневу топологію системи, виділивши ключові домени (Reader Service, Catalog Service тощо);
     - розробив логіку взаємодії компонентів (вибір синхронних REST-запитів та асинхронної обробки через RabbitMQ);
     - обрав конкретні технології для поліглотного зберігання даних (PostgreSQL, Elasticsearch, Redis, S3).
     Згенеровані за допомогою ШІ фрагменти програмного коду (наведені у Додатку В) були перевірені, модифіковані та інтегровані автором у звіт виключно для ілюстрації застосованих архітектурних принципів. Таким чином, ШІ було використано виключно як допоміжний інструмент для рутинних завдань, тоді як уся ключова бізнес-логіка та архітектурні рішення були сформовані автором самостійно.
     Перший фрагмент коду (див. додаток В) демонструє реалізацію REST-контролера управління закладками. Використання DTO дозволяє безпечно передавати дані між клієнтом та сервером, дотримуючись принципів клієнт-серверної архітектури та слабкого зв'язування.
     Другий фрагмент коду (див. додаток В) демонструє сервіс завантаження розділів. Цей компонент ілюструє інтеграцію з об'єктним сховищем S3 та використання шини повідомлень RabbitMQ для асинхронної обробки зображень, що є критично важливим для високонавантажених систем.
     Третій фрагмент коду (див. додаток В) ілюструє застосування системи кешування на базі Redis. Вона інтегрована в шар доступу до даних для мінімізації навантаження на PostgreSQL (стратегія Cache-Aside), що демонструє принцип Polyglot Persistence та оптимізацію продуктивності системи.

namespace SlashLIB.ReaderService.Services
{
    public class ChapterService : IChapterService
    {
        private readonly IMessageBus _bus;
        private readonly IStorageService _storage;

        public ChapterService(IMessageBus bus, IStorageService storage)
        {
            _bus = bus;
            _storage = storage;
        }

        public async Task UploadChapter(ChapterUploadRequest request)
        {
            // 1. Збереження оригіналів у S3
            var storagePath = await _storage.SaveAsync(request.Files);
            
            // 2. Публікація події для обробки зображень

            { 
                Id = request.Id, 
                Path = storagePath

        }
    }

37using Microsoft.EntityFrameworkCore;
using SlashLIB.CatalogService.DTOs;
using SlashLIB.CatalogService.Interfaces;
namespace SlashLIB.CatalogService.Services
{
    public class MangaCatalogService : IMangaCatalogService
    {
        private readonly ICacheService _cache;
        private readonly CatalogDbContext _db;

        public MangaCatalogService(ICacheService cache, CatalogDbContext db)
        {
            _cache = cache;
            _db = db;
        }

        public async Task<MangaDto> GetMangaAsync(int id)
        {
            var cacheKey = $"manga_{id}";
            
            return await _cache.GetOrSetAsync(cacheKey, async () => 
            {
                var manga = await _db.Manga
                    .Include(m => m.Chapters)
                    .FirstOrDefaultAsync(m => m.Id == id);
                    
                return manga?.ToDto();
            }, TimeSpan.FromMinutes(30));
        }
    }
