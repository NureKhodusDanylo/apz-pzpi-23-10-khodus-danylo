3 ОПИС ВИКОНАНОЇ РОБОТИ
      3.1 Загальні відомості про шаблони (патерни) проєктування
      Принципи та патерни проєктування - це типові, перевірені рішення поширених проблем під час розробки програмного забезпечення. Вони не є готовим кодом, а описують підхід до організації структури та взаємодії об’єктів. У контексті архітектури ПЗ ключову роль відіграють принципи SOLID. Останній принцип - Dependency Inversion - на практиці найчастіше реалізується через Dependency Injection. Такий підхід зменшує зв’язність коду, підвищує його гнучкість, спрощує Unit-тестування та полегшує підтримку великих систем.
      Використання патернів дозволяє розробникам спілкуватися єдиною технічною мовою та уникати "винайдення велосипеда". Традиційно шаблони проєктування поділяють на три великі категорії:
      1. Твірні (Creational): відповідають за безпечне та гнучке створення об'єктів.
      2. Структурні (Structural): допомагають формувати великі структури з окремих об'єктів без втрати гнучкості.
      3. Поведінкові (Behavioral): визначають алгоритми та способи взаємодії між об'єктами.
     У контексті побудови надійної архітектури програмного забезпечення ключову роль відіграють базові принципи об'єктно-орієнтованого дизайну, відомі під акронімом SOLID. Ці п'ять правил допомагають створювати системи, які легко підтримувати, розширювати та рефакторити.
      Особливе місце в сучасній розробці посідає останній принцип SOLID - Dependency Inversion Principle (DIP, Принцип інверсії залежностей). Згідно з ним, модулі верхнього рівня (бізнес-логіка) не повинні залежати від модулів нижнього рівня (робота з базою даних, мережею тощо); обидва рівні мають залежати від абстракцій (інтерфейсів).
      
      
      
    3.2 Опис обраного патерна
      Назва патерна: Dependency Injection (Ін’єкція залежностей)
      Тип шаблона (патерна): архітектурний (пов’язаний із породжувальним підходом)
      Основне призначення: передавати залежності об’єкту ззовні, а не створювати їх усередині класу. Об’єкт не використовує new для створення сервісів, а отримує готові реалізації через конструктор, сеттер або інтерфейс. Це робить систему більш гнучкою, тестованою та зручною для масштабування.
      Передумови застосування:
      1. Коли класи занадто жорстко пов’язані між собою (tight coupling), і зміна реалізації однієї залежності вимагає змін у багатьох місцях коду.
      2. Коли потрібно покривати код Unit-тестами: DI дозволяє легко підміняти реальні реалізації на mock/stub-об’єкти.
      3. Коли необхідно швидко змінювати реалізації сервісів (наприклад, джерело даних або логер) без зміни бізнес-логіки.
      4. Коли класи беруть на себе зайву відповідальність, самостійно створюючи й налаштовуючи залежності замість використання готових абстракцій.
      Переваги:
      1. Зменшує сильну зв’язність між компонентами системи та формує більш гнучку loosely-coupled архітектуру.
      2. Покращує тестованість коду завдяки простій підміні залежностей у модульних тестах.
      3. Спрощує розширення функціоналу: нові реалізації сервісів додаються без змін у клієнтському коді (Open/Closed Principle).
      4. Розділяє логіку створення об’єктів і логіку їх використання, що підвищує повторне використання коду.
      5. Дозволяє централізовано керувати конфігурацією та життєвим циклом об’єктів через інжектор або DI-контейнер.
      
      Недоліки:
      1. Збільшує кількість інтерфейсів і класів у проєкті, що робить структуру більш об’ємною.
      2. Для дуже простих задач може бути надлишковим рішенням (overkill).
      3. Ускладнює розуміння потоку виконання: не завжди очевидно, яка конкретна реалізація буде використана в момент запуску.
      4. Потребує додаткових знань для налаштування DI-контейнерів і фреймворків.
       3.3 Структура шаблона (патерна)
      Опис структурних частин:
      1. Client (Клієнт) - клас, який потребує зовнішніх залежностей для виконання своєї роботи.
      2. Service (Інтерфейс сервісу) - абстракція, що визначає контракт методів, необхідних клієнту.
      3. Injector / Container (Інжектор) - механізм, який створює конкретні реалізації сервісів та передає їх у Client.
      Опис взаємодії об’єктів
      1. Client визначає свою потребу в Service (зазвичай через параметри конструктора або сеттер).
      2. Service задає контракт, через який Client працює з будь-якою реалізацією залежності.
      3. Injector створює екземпляр конкретної реалізації сервісу (ConcreteService) та виконує впровадження залежності у Client.
      
      Передача залежності відбувається під час створення клієнта або через окремий метод встановлення сервісу.
      4. Завдяки цьому Client взаємодіє лише з абстракцією Service і не залежить від деталей створення об’єктів.
      5. Результат: код стає слабкозв’язним, легшим для тестування, модифікації та масштабування.
      UML-діаграма з поясненнями
         
Рисунок 1 - UML-діаграма
      UML-діаграма для Dependency Injection показує взаємодію Client, Service та ConcreteService, а також роль Injector. Client зберігає посилання на інтерфейс Service і викликає його методи. ConcreteServiceA та ConcreteServiceB реалізують цей інтерфейс. Injector (або DI-контейнер) створює потрібну реалізацію та передає її клієнту через конструктор або сеттер, реалізуючи інверсію контролю. Завдяки цьому клієнт залежить від абстракції, а процес складання об’єктів винесено назовні.
       3.4 Реалізація шаблона (патерна)
      Опис реалізації мовою програмування C#
      Основні етапи:
      1. Створення базових інтерфейсів
      
      Інтерфейси залежностей (наприклад, ICustomerRepository та ILogger) 
      задають єдиний контракт для роботи з даними та логуванням. Клієнтський клас не створює ці об’єкти самостійно, а працює через абстракції.
      Конструкторне впровадження: клас CustomerService отримує залежності через конструктор і зберігає їх у приватних полях для подальшого використання у бізнес-методах.
      Такий підхід прибирає жорстке зчеплення з конкретними реалізаціями та дозволяє легко замінювати сервіси без змін у коді клієнта (Див. додаток В.1).
      2. Реалізація конкретних класів
      Базовий інтерфейс IService визначає контракт методу DoSomething(), який реалізують конкретні сервіси.
      Клас ServiceImplementation реалізує IService і містить конкретну логіку. Це дає змогу додавати альтернативні сервіси без переписування коду.
      Setter Injection: залежність передається після створення клієнта через метод setService(...). Це корисно, коли сервіс може змінюватися динамічно (Див. додаток В.2).
      3. Демонстрація використання
      Service - базовий інтерфейс для залежностей. ConcreteServiceA та ConcreteServiceB - конкретні реалізації сервісу. Client зберігає посилання на Service і виконує роботу через інтерфейс, не знаючи деталей конкретного класу. main() виконує роль інжектора: створює ConcreteServiceA, передає його в Client, викликає методи клієнта. Після цього в main() можна підмінити залежність на ConcreteServiceB, і поведінка клієнта зміниться без втручання в його код (Див. додаток В.3).

0:12 Вступ

0:52 Обраний патерн, його тип та призначення
Розглядається головна ідея патерну — відмова від ключового слова new всередині класів при створенні залежностей. Замість цього об'єкт отримує вже готові сервіси ззовні (наприклад, через конструктор).

1:22 Передумови застосування
Описуються ситуації, в яких доцільно використовувати DI: жорстка зв'язність між класами, необхідність покриття коду unit-тестами за допомогою мок-об'єктів та потреба в легкій заміні реалізацій сервісів без зміни бізнес-логіки.

1:58 Переваги використання патерну
Розглядаються основні плюси: отримання гнучкої архітектури без жорстких зв'язків, значне спрощення тестування та розділення логіки створення об'єктів для їх повторного використання (Reusability).

2:23 Недоліки використання патерну
Аналізуються можливі мінуси: зростання загальної складності системи, поява великої кількості додаткових інтерфейсів та ускладнення розуміння потоку виконання програми, що може бути зайвим (Overkill) для маленьких проєктів.

2:51 Опис структурних частин
Пояснюються три ключові елементи патерну: Client (основний клас із бізнес-логікою), Service (інтерфейс, який описує залежність) та Injector (механізм або контейнер, що створює реалізацію і передає її клієнту).

3:20 Опис взаємодії об'єктів
Описується процес взаємодії: клієнт вказує потребу в сервісі, інжектор самостійно створює конкретну реалізацію цього сервісу та передає її клієнту, який просто використовує готовий об'єкт, не знаючи деталей його створення.

3:48 UML - діаграма
Демонструється UML-діаграма патерну, яка формально відображає незалежність клієнта від конкретних реалізацій та показує відповідальність інжектора за створення та зв'язування об'єктів.

4:17 Основні етапи: Впровадження через конструктор
Розглядається класичний і найпопулярніший спосіб впровадження залежностей (Constructor Injection) на прикладі реального коду класу CustomerService, який вимагає передачі інтерфейсів через параметри конструктора.

4:59 Основні етапи: Впровадження через властивість (Setter Injection)
Демонструється інший спосіб передачі залежностей через публічну властивість класу (setter). Це дозволяє замінювати сервіси вже після того, як об'єкт клієнта був створений.

5:30 Основні етапи: Реальне використання DI-контейнера (Клієнт)
Показується реальне використання патерну у сучасному проєкті, де метод main виконує роль інжектора (DI-контейнера), який реєструє всі інтерфейси та автоматично збирає залежності під капотом.

6:08 Висновок
Підсумовується значення патерну Dependency Injection, який перетворює монолітну програму на гнучку модульну систему. Це дозволяє великим командам розробників паралельно створювати масштабований код, який легко тестувати та підтримувати.

Запити до ШІ : Створи три приклади коду на C# для демонстрації патерну Впровадження Залежностей (Dependency Injection). Кожен приклад повинен показувати різні підходи до реалізації:
1. Напиши приклад сервісу CustomerService на мові C#, який використовує Constructor Injection (впровадження через конструктор). Сервіс повинен реалізовувати інтерфейс ICustomerService та залежати від ICustomerRepository і ILogger. Додай перевірку залежностей на null та асинхронний метод GetCustomerById, який логує процес отримання даних.
2. Продемонструй приклад Setter Injection (впровадження через властивість) на C#. Створи інтерфейс IService, його реалізацію та клас Client, у якого є публічна властивість для цієї залежності. Покажи в методі Main, як вручну створити об'єкт клієнта та встановити залежність після його ініціалізації.
3. Напиши приклад налаштування стандартного DI-контейнера (Inversion of Control container) у консольному додатку .NET за допомогою бібліотеки Microsoft.Extensions.DependencyInjection. Покажи, як зареєструвати сервіси з різними життєвими циклами (Transient та Scoped), як побудувати ServiceProvider та отримати з нього головний об'єкт додатка ClientApp з уже впровадженими залежностями.

46using System;
using CustomerApp.Core.Interfaces;
using CustomerApp.Core.Models;
using Microsoft.Extensions.Logging;

namespace CustomerApp.Core.Services
{
    public class CustomerService : ICustomerService
    {
        private readonly ICustomerRepository _repository;
        private readonly ILogger<CustomerService> _logger;

        // Constructor with Dependencies Injected
        public CustomerService(ICustomerRepository repository, ILogger<CustomerService> logger)
        {
            _repository = repository ?? throw new ArgumentNullException(nameof(repository));
            _logger = logger ?? throw new ArgumentNullException(nameof(logger));
            _logger.LogInformation("CustomerService initialized with {Repository}", _repository.GetType().Name);
        }

        public async Task<Customer> GetCustomerById(int id)
        {
            _logger.LogInformation("Fetching customer with ID: {Id}", id);
            var customer = await _repository.GetByIdAsync(id);

            if (customer == null)
            {
                _logger.LogWarning("Customer with ID {Id} not found.", id);
                return null;
            }

            return customer;
        }
    }
}

38namespace SetterInjectionDemo
{
    // 1. Dependency Interface
    public interface IService {
        void DoSomething();
    }

    // 2. Implementation
    public class ServiceImplementation : IService {
        public void DoSomething() {
            Console.WriteLine("Service done.");
        }
    }

    public class Client {
        // 3. Setter Injection: The dependency property. It is set after instantiation.
        public IService Service { get; set; }

        public void PerformAction() {
            Service?.DoSomething();
        }
    }

    class Program
    {
        static void Main(string[] args)
        {
            // 4. Instantiate Client
            Client client = new Client();
            
            // 5. Manually inject the dependency.
            client.Service = new ServiceImplementation();
            client.PerformAction();
        }
    }
}

38using Microsoft.Extensions.DependencyInjection;
using MyApp.Interfaces;
using MyApp.Services;

namespace MyApp
{
    class Program
    {
        static void Main(string[] args)
        {
            //  1. Створюємо колекцію сервісів (DI-контейнер)
            var services = new ServiceCollection();

            // 2. Реєструємо наші залежності (Інжектор)
            services.AddTransient<IEmailService, EmailService>();
            services.AddScoped<IUserRepository, UserRepository>();

            // Реєструємо нашого Клієнта
            services.AddTransient<ClientApp>();

            // 3. Будуємо провайдер сервісів
            var serviceProvider = services.BuildServiceProvider();

            // 4. Отримуємо готового Клієнта з усіма впровадженими залежностями
            var app = serviceProvider.GetRequiredService<ClientApp>();

            // 5. Запускаємо програму
            app.Run();
        }
    }
}
